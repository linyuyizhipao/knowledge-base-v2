**版本**: 1.0 | **更新**: 2026-04-27

## 一、复杂度分级

| 复杂度 | 触发条件 | 编排方式 | 示例 |
|--------|---------|---------|------|
| **简单** | 单文件改动，<50行，逻辑明确 | 直接写，不用 sub-agent | 加个字段、修个 null check、改命名 |
| **中等** | 多文件，单一项目内 | Explore（调研）→ 编写代码 → code-reviewer（审查） | 新增 API + Service + DAO |
| **复杂** | 跨项目 / 架构级改动 | Plan（设计）→ Explore（调研）→ general-purpose（实现）→ code-reviewer（审查） | 369 跨项目改造、多应用隔离 |

## 二、Agent 类型

### Explore（快速只读搜索）

- **能力**: 文件查找、grep、Read、WebFetch、WebSearch（只读，不可编辑）
- **用途**: 快速定位代码、查找符号定义、调研现有模式
- **何时用**: 需要"xxx 在哪里定义的？"、"哪个文件处理了这个事件？"这类问题时
- **何时不用**: 已经知道文件路径时直接用 Read；需要 grep 时直接用 Bash

### Plan（方案设计）

- **能力**: 读取、搜索、分析（不可编辑文件）
- **用途**: 设计实现方案、识别架构权衡、拆分任务步骤
- **何时用**: 需要"这个需求怎么拆？"、"有哪些实现方案？哪个更好？"时
- **何时不用**: 需求明确、方案唯一时跳过

### general-purpose（通用实现）

- **能力**: 所有工具（读写、Bash、Agent）
- **用途**: 实际编写代码、创建文件、运行测试
- **何时用**: 需要并行实现多个独立模块时（比如同时写 3 个 Service）
- **何时不用**: 代码量小、主上下文可以一次完成时跳过

### code-reviewer（代码审查）

- **能力**: 所有工具
- **用途**: 审查代码与计划的一致性、检查规范、发现潜在问题
- **何时用**: 一个阶段完成后需要独立视角检查
- **何时不用**: 极小改动（单行修复）可以跳过

## 三、编排模式

### 模式 A：简单任务（直接编写）

```
用户给需求
    ↓
读取知识
    ↓
直接编写代码
    ↓
提交
```

### 模式 B：中等任务（调研 + 实现 + 审查）

```
用户给需求
    ↓
Explore: "查找现有的类似实现和模式"
    ↓
读取调研报告
    ↓
编写代码（主上下文）
    ↓
code-reviewer: "审查这个实现"
    ↓
修复问题 → 提交
```

### 模式 C：复杂任务（设计 + 调研 + 实现 + 审查）

```
用户给需求
    ↓
Plan: "设计实现方案，识别关键文件"
    ↓
读取设计方案
    ↓
Explore: "查找相关代码和现有模式"
    ↓
general-purpose: "实现模块 A"（并行）
general-purpose: "实现模块 B"（并行）
    ↓
读取实现结果
    ↓
code-reviewer: "审查整个实现"
    ↓
修复问题 → 提交
```

## 四、决策流程图

```
来了一个新任务
    ↓
涉及几个项目？
    ├─ 0-1个，且改动 < 50行 → 模式 A（直接写）
    ├─ 0-1个，且改动 > 50行 → 模式 B（调研+实现+审查）
    └─ 2个以上，或架构级改动 → 模式 C（设计+调研+实现+审查）
```

---

## 五、用户覆盖

用户可以通过以下方式覆盖默认策略：

- **"直接用 sub-agent"** — 强制走模式 C，即使任务简单
- **"不用 sub-agent"** — 强制模式 A，即使任务复杂
- **"用 Explore 查一下 xxx"** — 手动指定特定 agent
- 没有明确指令时，按复杂度分级自动选择

## 六、与 AI_WORKFLOW 的关系

Sub-agent 编排在 `AI_WORKFLOW.md` 的 **步骤4（实施开发）** 和 **步骤5（合入与部署）** 中触发：

| AI_WORKFLOW 步骤 | 触发的 agent 编排 |
|------------------|------------------|
| 步骤 0：创建功能分支 | 不适用（简单 git 命令） |
| 步骤 1：需求分析 | 不适用（在主上下文中完成） |
| 步骤 2：知识检索 | Explore agent（中/复杂任务） |
| 步骤 3：方案设计 | Plan agent（复杂任务） |
| 步骤 4：实施开发 | general-purpose agent（复杂任务的并行实现） |
| 步骤 5：合入与部署 | code-reviewer agent（审查环节） |
| 步骤 6：知识沉淀 | 不适用（在主上下文中完成） |
